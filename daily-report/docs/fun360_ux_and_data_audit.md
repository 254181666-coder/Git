# Fun360 后台体验与数据体系审计

观察目标：理解 Fun360 为什么对管理不友好，并提炼自研轻量系统的改进方向。

观察范围：

- 汇总报表
- 明细报表
- 自定义报表
- 支付明细
- 财务对账入口
- 私人订制报表

## 总体判断

Fun360 当前后台的主要问题不是“没有数据”，而是：

1. 数据没有经过管理视角的二次清洗。
2. 报表太多，字段太细，缺少统一指标口径。
3. 同类报表分散在不同菜单，用户需要自己拼口径。
4. 下载方式不统一，管理层难以稳定复用。
5. 筛选器、日期口径、门店口径在不同页面表现不一致。
6. 标准报表、自定义报表、私人订制报表混在一起，用户不知道哪个才是准口径。

这说明我们自研系统不应该复制 Fun360 的页面数量，而应该做“少页面、强口径、可追溯”的经营系统。

## 菜单冗余

### 汇总报表 vs 自定义报表

同样是收入相关数据，至少分散在：

- 汇总报表 / 收入汇总
- 自定义报表 / 营业分析
- 自定义报表 / 财务明细
- 自定义报表 / 【私人订制】收入表
- 自定义报表 / 【私人订制】日营业数据表
- 明细报表 / 支付明细

问题：

- 管理者不知道“收入”到底应该看哪个页面。
- 标准收入、经营收入、财务到账收入、支付收入混在一起。
- 不同页面的字段名看似接近，但口径不同。

自研优化：

- 合并成一个“经营收入”模块。
- 顶部先给管理口径指标：营业总收入、净收入、实收、实际到账、储值、团购、房费、商品。
- 下钻才进入支付方式、收入类型、订单明细。

### 商品相关页面过多

商品数据分散在：

- 商品销售排行
- 商品销售明细
- 自定义报表 / 商品销售分析
- 自定义报表 / 商品赠送分析
- 私人日报里的超市收入
- 收入汇总里的商品类型占比

问题：

- 销售、套餐、套餐子物品、赠送、例送、分摊金额混在不同表。
- 商品分类没有被统一成管理层的大类。
- 商品销售金额和商品经营贡献容易混淆。

自研优化：

- 合并成一个“商品经营”模块。
- 固定三层视图：商品大类、商品、订单明细。
- 明确区分普通销售、套餐、套餐子物品、赠送、退款。

### 支付与收入混杂

支付相关数据分散在：

- 支付汇总
- 支付明细（不含储值）
- 支付明细（含储值）
- 储值支付
- 包厢支付
- 卡券活动支付
- 营销预定收款
- 预订
- 私人订制收入表

问题：

- 支付方式被当成收入项目使用。
- 储值支付、会员本金消费、会员赠金消费、实际到账之间关系不清楚。
- 管理者难以判断“今天到底到账多少钱”。

自研优化：

- 建立 `clean_payment_event`。
- 支付方式、收入类型、业务来源三个维度分离。
- 页面只显示三个核心数：应收、实收、到账。

## 下载方式不统一

已观察到至少三种下载/导出形态：

1. 标准明细页：按钮名为 `导出Excel`。
2. Redash 小组件：每个小组件右上角一个 `下载`。
3. 私人订制/自定义报表：多个图表块各自有 `下载`，有些页面还带刷新、编辑布局。

问题：

- 用户不知道下载的是整页、当前图表、还是某个查询结果。
- 同一个页面可能有多个下载按钮。
- 下载结果是否遵循当前筛选条件不明显。
- 管理动作依赖人工下载和人工拼表。

自研优化：

- 所有页面统一导出入口：`导出当前视图`、`导出明细`、`导出对账包`。
- 导出文件命名统一：`业务模块_日期范围_门店_生成时间.xlsx`。
- 所有导出必须记录筛选条件和指标口径。
- 管理报表尽量不要求人工下载，而是内置汇总、对账、异常提示。

## 筛选方式不统一

已观察到的筛选差异：

- 有些页面使用“展开筛选/收起筛选”。
- 有些页面直接显示表单筛选。
- 日期有“营业日期”“时间”“开房时间”等不同名称。
- 时间范围有的按日期，有的按 `08:00-08:00`，有的按 `09:00-09:00`。
- 门店默认值有时是全部，有时是总部，有时是单店。

问题：

- 口径容易错。
- 管理者不知道当前数据是否按营业日计算。
- 总部、系统自动开房、外卖包、超市包等异常记录容易混入分析。

自研优化：

- 统一筛选栏：营业日、门店、业务主题、对比周期。
- 默认营业日统一 `08:00 ~ 次日 08:00`，特殊报表必须标注。
- 总部和系统自动记录默认排除，但支持在高级筛选中查看。
- 每张图表显示当前口径标签。

## 数据没有清洗直接展示

典型表现：

- 商品分类过细，管理层看不到稳定大类。
- 支付方式、支付来源、收入类型混在一起。
- 系统自动开房、总部、外卖包、超市包、大厅包直接出现在表格里。
- Redash 小组件直接暴露查询结果，缺少业务解释。
- 退款/冲减没有作为独立业务事件清晰呈现。

自研优化：

- Raw 层保留原始字段。
- Clean 层清洗业务事件。
- Mart 层只暴露管理指标。
- 所有异常记录进入“数据质量/异常记录”模块，而不是混进日报。

## 推荐的自研页面结构

### 1. 经营总览

面向老板/管理层。

展示：

- 总收入
- 净收入
- 实收
- 实际到账
- 储值
- 团购
- 商品
- 房费
- 开台数
- 客单价
- 异常提醒

特点：

- 不展示原始明细字段。
- 所有指标可下钻。

### 2. 收入分析

面向财务和运营。

视图：

- 按门店
- 按收入类型
- 按支付方式
- 按营业日趋势
- 按异常项：挂账、短款、招待、赠免、退款

### 3. 商品经营

视图：

- 大类销售
- 商品排行
- 门店对比
- 套餐/赠送/退款分析
- 低效商品

### 4. 会员与储值

视图：

- 储值金额
- 储值人数
- 首充/复充
- 本金/赠金
- 储值后消费

### 5. 团购与预订

视图：

- 团购订单
- 团购净额
- 退款
- 核销/到店
- 预订到店率

### 6. 数据质量中心

给我们内部使用。

展示：

- 同步是否成功
- raw 行数
- clean 行数
- mart 行数
- 会员画像覆盖率
- 开台详情覆盖率
- 未归类商品
- 对账差异

## 推荐的系统原则

1. 页面少于 Fun360，但每个页面更明确。
2. 每个指标必须能下钻到明细。
3. 每个指标必须有口径说明。
4. 下载不是核心功能，分析和异常提醒才是核心功能。
5. 默认展示管理口径，高级用户再看明细。
6. 所有原始字段不直接进入管理页面，必须先经过 Clean 层。
7. 每个导出都必须包含筛选条件和口径版本。

## 开发落地顺序

1. 建 Clean 表，不再让报表直接查 raw。
2. 建 Mart 表，先覆盖经营总览。
3. 建数据质量中心，先服务我们自己。
4. 做对账差异表，逐日减少和 Fun360 私人订制报表的差异。
5. 做经营总览页面。
6. 做商品经营页面。
7. 做会员储值和团购预订页面。
8. 最后再考虑替代或绑定收银机。
