# Fun360 后台观察与自研系统启发

观察时间：2026-05-20  
入口：`https://ktv.fun360.cn/#/custom/data_analysis`

## 后台结构

Fun360 后台主要分成几类：

- 汇总报表：收入汇总、开台汇总、趋势汇总、收入目标汇总、营运日报。
- 明细报表：开房套销售明细、商品销售排行、订单明细、会员报表、商品销售明细、卡券明细、支付明细、提成报表等。
- 自定义报表：财务明细、营业分析、商品销售分析、会员消费明细、商品赠送分析，以及多张私人订制报表。

一个重要发现：部分自定义报表是嵌入 Redash 查询。说明 Fun360 本身也在用“底层业务数据 + 报表查询层”的模式，而不是所有指标都由交易系统页面直接生成。

- 储值本金/赠送、会员本金/赠送、购卡本金/赠送要分开。

### 商品销售分析

路径：

- `#/custom/product_summary`

关键字段：

- 品牌
- 门店
- 商品名称
- 商品类别
- 区域
- 时间范围
- 类别名称
- 成本
- 金额（折后）
- 利润
- 利润率
- 数量
- 金额
- 销售数量：小计、销售、套餐、套餐子物品、例送子物品
- 销售金额：小计、销售、套餐、套餐子物品、套餐子物品分摊、例送子物品
- 赠送数量：小计、赠送、套餐、套餐子物品
- 赠送金额：小计、赠送、套餐、套餐子物品

对我们的启发：

- 商品销售要区分普通销售、套餐、套餐子物品、例送、赠送。
- 套餐子物品需要有“分摊金额”字段，否则商品利润和大类销售会失真。
- 商品事实表建议保留 `sale_type`、`is_package_item`、`allocated_amount`、`gift_amount`。

- 线上团购应收
- 日单批消费

- 18点前维护
- 18点-24点待客

- 00点后维护
- 晚场待客最高峰台数
- 晚场待客最高峰时点

对我们的启发：

- 你们真正关心的是“经营日报口径”，不是 Fun360 标准订单口径。
- 应该建设 `mart_daily_store_revenue`，而不是让日报直接查订单明细。
- 需要保留时段拆分：18点前、18点-24点、00点后。
- “维护”是业务特有指标，应作为独立字段，不要混入待客。

- 线上支付：抖音、美团、快手
- 其他支付：其他
- 销售折扣：赠免、招待
- 挂账
- 短款
- 储值卡：现金、银联、美团、抖音、快手、其他
- 次卡：现金、银联、美团、抖音、快手、其他
- 储值卡赠金
- 净收入
- 营业外收入
- 实际到账的净额

对我们的启发：

- 财务收入表天然是二维结构：项目类型 × 支付方式。
- 我们需要一个 `fact_payment` 或 `clean_payment_event`，把支付方式标准化。
- “营业总收入”“净收入”“实际到账净额”是三个不同指标，不能互相替代。

- `消费金额`、`优惠金额`、`已付金额`、`未付金额`、`计入收入金额` 需要全部保留，不能只取一个金额。
- 系统自动开房、总部、外卖包、超市包、大厅包这类记录需要规则过滤或单独标记。

## 对自研轻量系统的启发

Fun360 的特点是功能全、字段多、报表复杂。我们未来更适合做一套轻量但口径清楚的系统。

核心不是复制所有功能，而是抽出最稳定的业务对象：

- 门店
- 包厢/台
- 开台/关台
- 订单
- 商品明细
- 支付流水
- 退款/冲减
- 会员
- 储值
- 团购/券核销
- 预订
- 赠送/招待
- 挂账/销账
- 营业日汇总

## 下一步开发建议

1. 先把 Fun360 私人订制报表字段映射成我们的 Mart 指标字典。
2. 把 `store_daily` 视为旧版业务基准，不再盲目依赖。
3. 从 `clean_payment_event` 和 `clean_order_item` 开始建设 Clean 层，因为收入和商品差异最大。
4. 对总部、系统自动开房、外卖包、超市包、大厅包建立过滤/标记规则。
5. 把“营业总收入、净收入、实际到账净额”定义成三个独立指标。
