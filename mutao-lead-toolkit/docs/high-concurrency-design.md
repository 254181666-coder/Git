# 高并发设计原则

## 设计结论

系统必须按高并发设计，但不能把“高并发”理解成对平台评论区高频自动回复。

正确边界是：

- 系统内部高并发：线索导入、去重、意图识别、任务运行、账号池、审计、多人协作都要能承载高吞吐。
- 平台动作低频可控：每个授权账号的回复、采集、检测必须排队、限流、可暂停、可审计。
- 人工确认优先：AI 只生成建议和待办，真实回复仍由人确认后执行。

## 当前 MVP 限制

当前版本是本地单机 MVP：

- 数据主要存储在 JSON 文件。
- Node 单进程提供本地服务。
- 适合演示、单客户试用、小团队验证业务流程。

它不是最终高并发生产架构。真实并发上来前，必须把核心数据和任务运行迁到可扩展架构。

## 生产架构方向

1. API 层无状态化
   - API 服务不保存运行状态。
   - 多实例横向扩容。
   - 会话、锁、队列状态放到 Redis 或数据库。

2. 数据库升级
   - 从 JSON 升级到云端 MySQL。
   - 关键表包括：tenants、users、leads、source_samples、discovery_candidates、reply_accounts、account_credentials、task_runs、audit_logs、rate_limits。
   - 所有业务表必须带 tenant_id，并建立 tenant_id + 业务索引，确保客户数据隔离和高并发查询效率。
   - 所有写入使用事务，避免多人同时操作造成数据错乱。

3. 队列化任务
   - 评论导入、意图识别、去重合并、通知、授权检测全部进入任务队列。
   - 建议使用 Redis Queue / BullMQ / RabbitMQ 一类队列。
   - 任务必须支持重试、失败原因、幂等键、取消和恢复。

4. 账号级限流
   - 每个账号维护独立配额：每日回复数、每小时回复数、最短间隔、异常暂停。
   - 同一账号的真实回复动作必须串行排队。
   - 不同账号可以并行，但仍受平台、客户、租户级总限流约束。

5. 幂等和去重
   - 评论样本使用来源平台、作品 ID、评论 ID、作者标识生成幂等键。
   - 线索生成使用 dedupeKey，重复命中只合并来源和跟进记录。
   - 回复记录使用 leadId + accountId + 原评论定位 + 时间窗口防重复。

6. 审计不可丢
   - 所有账号绑定、检测、回复、清除绑定、导出、任务运行必须写审计。
   - 审计日志使用 append-only 模式。
   - 高并发下审计可以异步写入，但不能无记录执行关键动作。

7. 背压和降级
   - AI 意图识别要有每日预算、队列长度上限和缓存。
   - 队列积压时优先处理高意向和已授权来源。
   - 外部平台不可用时，不丢数据，只暂停任务并记录失败原因。

## 并发等级规划

### 阶段 1：本地验证

- 目标：单人或小团队验证业务闭环。
- 数据：JSON。
- 并发：低。
- 重点：流程正确、字段完整、安全边界明确。

### 阶段 2：单客户生产

- 目标：一个客户团队多人使用。
- 数据：MySQL；单机轻量版可用 SQLite 过渡。
- 并发：几十个账号、几千到几万条线索。
- 重点：事务、账号限流、队列任务、定时备份。

### 阶段 3：多客户 SaaS / 私有化集群

- 目标：多租户、多团队、多账号矩阵。
- 数据：MySQL + Redis + 对象存储。
- 并发：多实例 API、多 worker、多队列。
- 重点：租户隔离、水平扩容、监控告警、审计留存、灾备。

## 后续开发要求

从现在开始新增模块时，必须遵守：

- 不把长任务直接塞进 HTTP 请求里执行。
- 不在内存里保存关键业务状态，除非只是临时会话。
- 所有写入要考虑幂等和重复提交。
- 所有账号动作都要先过账号限流。
- 所有平台动作都要可审计、可暂停、可回放状态。
- 不为了高并发牺牲风控边界，不做自动批量触达。

云端多租户迁移说明见 `docs/cloud-multi-tenant-migration.md`。
