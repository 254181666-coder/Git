# 验证码与风控部署规划

更新时间：2026-06-04

## 原则

- 不做验证码绕过、不接入打码绕过服务。
- 验证码、登录校验、回复过频都视为平台风控信号，系统只做识别、暂停、提醒、留痕和人工恢复。
- 账号维度和平台维度都要能独立暂停，避免一个账号触发风控后拖垮整个平台任务。
- 用户完成平台页面内的人工校验后，再由系统继续执行剩余队列。

## 明日团队测试口径

1. 小批量验证：每个平台先用 1-2 个账号、单轮 10-20 条、默认总量 200 的边界测试。
2. 观察风控：记录验证码、回复太频、登录失效、输入失败、找不到回复入口这几类原因。
3. 人工接管：出现验证码时不刷新、不重开任务，先在当前外部浏览器里完成校验。
4. 恢复执行：校验完成后回到工具包，使用继续或二次批量回复处理未成功线索。
5. 复盘归因：只统计命中评论，区分已成功、未成功、重复跳过、风控暂停。

## 系统处理流程

```text
任务执行
  -> 检测页面状态
  -> 命中验证码/登录校验/回复太频
  -> 暂停当前账号队列
  -> 前端提示处理原因和账号
  -> 用户在真实浏览器完成人工校验
  -> 用户点击继续或执行二次批量
  -> 继续处理未成功线索
```

## 前端需要展示

- 当前账号、平台、作品链接。
- 暂停原因：验证码、登录校验、回复太频、页面异常、输入失败。
- 当前队列进度：已浏览、命中、成功、失败、重复跳过。
- 操作按钮：暂停任务、结束任务、继续任务、二次批量回复未成功。
- 风控提示：建议等待时间、当前账号今日回复数、单轮回复数。

## 数据留痕

每次触发风控都保留：

- 平台、账号、作品链接。
- 触发时间。
- 风控类型。
- 当时的命中评论和回复话术。
- 是否人工处理。
- 后续是否恢复成功。

## 部署边界

- 单账号单轮：10-20 条。
- 默认单任务总量：200 条。
- 同作品同评论：成功后永久筛重，不连续重复回复。
- 同平台多账号：支持轮换，不限制账号池规模。
- 双平台并行：可以并行，但建议每个平台独立队列、独立暂停、独立复盘。

## 上线建议

- 明天团队测试先用本地安装包，不直接开放公网多人使用。
- 测试通过后再部署公网服务端，公网只管理任务、账号、复盘和队列，不托管平台验证码处理。
- 浏览器校验仍发生在执行机器的真实浏览器中，由账号负责人处理。
- 公网部署后要加入操作日志和账号健康看板，便于判断是不是某个账号、某个平台或某类作品触发风控。
