# 2026-05-25 进度存档

## 存档背景

今天的主线不是继续堆功能，而是根据需求方演示、截图和真实测试反馈，把产品方向重新收敛到一个更清楚的目标：

从公开评论里采集“人”的信息，沉淀成线索人资产，再让业务人员或官方号围绕这些人做可控跟进。

当前项目不再把“导入线索”作为主要路径。真实业务里客户主动留下微信、手机号或表单的概率很低，所以现阶段最重要的是把评论人、抖音号/ID、IP/属地、评论内容、来源作品采集完整，并能快速对应回平台页面继续跟进。

## 当前产品主链路

1. 创建采集任务
   - 行业入口可以先用行业目标词找方向。
   - 作品、作者、直播、同城等初采集阶段不强制触发词。
   - 默认先全量采集当前作品页可见评论，采后再进入识别。

2. 评论进入评论池
   - 评论池是原始数据层。
   - 未命中识别词的评论也保留，不再因为没有触发词就丢弃。
   - 采后识别词只用于生成候选线索，不决定是否入库。

3. 评论沉淀为线索人资产
   - 核心资产不是评论本身，而是评论背后的用户。
   - 线索人资产保留：用户名、抖音号/ID、IP/属地、评论内容、采集渠道/来源作品、用户主页链接。
   - 去重逻辑调整为“同人同内容”，不再只按同作品同内容去重。

4. 线索进入跟进
   - 人工可打开来源作品、打开用户主页、复制评论定位信息。
   - 员工号/粉丝号可用于人工回复、人工反关、人工私信并回到系统标记。
   - 官方号后续走“授权后小范围受控自动”，不是无上限批量触达。

## 当前已知问题

1. 平台页面真实采集仍依赖授权窗口可见页面
   - 当前不是平台后台全量接口采集。
   - 能采到多少，取决于授权窗口是否打开了具体作品详情页、评论区是否展开、平台是否加载更多评论。
   - 系统现在会记录采集轨迹，方便判断问题发生在页面状态、滚动翻页，还是识别解析。

2. 抖音授权存在闪退/反复退出风险
   - 已调整 User-Agent 和中文语言请求头，降低 Electron 内置窗口被误判概率。
   - 但平台侧“操作频繁”、扫码异常、登录后退出属于平台风控，系统不做绕过。
   - 需要在 Windows 常用网络和测试账号上继续验证。

3. BitBrowser 链接通知仍需继续排查
   - 用户反馈浏览器不停弹“需要使用新应用打开 bitbrowser 链接”。
   - 当前方向应尽量减少外部协议依赖，优先使用应用内置授权窗口。
   - 若需求方要求继续接 BitBrowser，需要单独确认协议、安装检测和失败兜底。

4. 复制作品链接仍需真实平台复测
   - 已支持从抖音搜索弹窗 `modal_id` 归一到作品链接。
   - 但用户反馈复制时会出现小白框、提示不清晰。
   - 后续应把复制结果直接显示在页面上，并提供“已复制的链接预览”和“填入采集任务”按钮。

5. 评论内容完整性还要继续压测
   - 已修正一部分 `@别人` 误提取和 `...` 空内容问题。
   - 但抖音虚拟列表、折叠回复、表情评论、长评论截断仍需要更多真实作品测试。

6. 页面自适应仍要继续优化
   - 线索列表已改表格，但大量字段下仍可能需要横向或纵向自适应。
   - 后续应做固定列、紧凑行高、评论内容展开/收起和操作列常驻。

## 当前产品判断

1. “评论池识别”更像中间层，不应该让业务人员长期停留在这里。
   - 业务人员真正需要的是线索人资产列表和跟进动作。
   - 评论池可以作为运营或管理员查看原始数据的地方。

2. “候选”不是失败。
   - 候选 0 只代表没有进入识别规则，不代表没采到评论。
   - 所以页面应优先展示“采集到多少评论、沉淀多少人资产”，再展示“识别出多少候选”。

3. “打开来源”必须拆清楚。
   - 打开评论来源：打开原作品/原评论场景。
   - 打开用户主页：打开评论人的主页。
   - 处理这条线索：进入本系统跟进视图。
   - 官方号私信：调用官方号授权能力或打开官方号操作窗口。

4. 官方号自动私信是关键可操作点。
   - 需求方明确认为官方号授权后可以做私信回复，不一定需要接 API。
   - 下一轮应围绕官方号授权窗口、首轮私信、二次私信、发送额度和发送记录做产品设计。

## 下一轮需要问需求方的问题

1. 平台优先级
   - 先只做抖音，还是抖音 + 小红书 + 快手同时做？

2. 官方号能力
   - 官方号是否已经具备私信权限？
   - 首轮私信和二次私信分别有什么平台限制？
   - 是否允许自动发送，还是必须人工确认？

3. 数据字段
   - 线索人资产最小必填字段是否就是：用户名、ID、IP/属地、评论内容、来源作品？
   - 是否需要头像、粉丝数、关注数、主页简介、作品数？

4. 采集规模
   - 单作品期望采集多少条？
   - 单账号每天可接受多少采集动作？
   - 小范围触达的上限是 20、50、100，还是按账号健康度动态调整？

5. 跟进动作
   - 员工号人工反关、人工私信是否作为默认流程？
   - 官方号自动私信是否作为高级流程？
   - 是否需要给销售分配线索和统计转化？

## 交接提醒

会议后如果要做大调整，建议不要直接在现有页面继续补丁式修改。优先按新共识重画主导航和主链路：

- 今日工作台
- 采集任务
- 线索人资产
- 官方号/员工号账号池
- 跟进与触达
- 运行日志/审计

现有“评论池识别、客户信息采集、客户跟进、回复账号池”等能力可以保留为底层能力，但前台操作应更像业务流程，而不是功能堆叠。
