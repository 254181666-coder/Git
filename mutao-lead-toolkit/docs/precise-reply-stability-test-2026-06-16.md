# 精准评论回复稳定性测试记录

更新时间：2026-06-16

## 测试目标

验证抖音和小红书「关键字精准回复」在真实授权页面中的稳定性。每轮测试必须拆分记录：

- 评论扫描：是否能打开作品、展开评论区、滚动并采集目标数量。
- 目标定位：是否能找到命中关键词的评论，并匹配评论作者/评论内容。
- 回复入口：是否能点击命中评论下方的「回复」或消息气泡。
- 回复框保持：是否出现并保持 `回复@目标用户` 或平台等价回复态。
- 写入结果：回复文案是否进入正确回复框，而不是作品主评论框。
- 发送结果：仅在明确勾选允许真实发送时验证；否则只记录「已写入待发送」。

## 安全边界

- 默认不勾选「允许真实发送」，先做定位和写入诊断。
- 未确认目标回复框时，不强制写入，不坐标补点，不刷新重试到丢失当前评论。
- 抖音必须确认 `回复@目标用户` 上下文，小红书必须确认刚点击回复入口附近的评论上下文。
- 真实发送测试每轮最多 1-3 条，失败立即停，不做连续冲量。

## 当前本地回归结果

- `node --check server.js`：通过。
- `GET /api/diagnostics`：通过，可读取版本、数据文件、授权、最近任务记录。
- `GET /api/state`：通过，可读取账号池、线索、任务、最近失败详情。
- 最近抖音失败高频原因：
  - 已定位回复元素，但回复框未稳定出现或未保持。
  - 回复框出现后，原生输入没有进入 `回复@某人` 编辑器。
  - 偶发 `Runtime.evaluate` 调用超时。

## 本轮代码调整

- 抖音点击命中评论「回复」后，等待 `回复@某人` 输入框出现的轮询窗口从约 1 秒延长到约 2.28 秒。
- 失败原因增加实际等待毫秒数，便于判断是页面慢、回复框未出现，还是后续写入失败。
- 第三轮肉眼观察确认草稿已快速进入回复框；后续修正重点从“写入失败”转为“写入成功后的活动回复编辑器验证”。
- 第四轮已验证活动回复编辑器复查生效：诊断接口正确记录 `已挂载 1`，不再把已写入草稿误判为失败。
- 第七轮已完成真实发送小测：按回车发送成功，接口记录 `sent=1 / failed=0`，输入框检测文本不再重复。
- 成功发送后已固定为继续扫描当前作品：达到本批回复上限时不再继续发送，但仍滚动读取评论到底、沉淀命中评论，当前作品完成后才切换下一条作品。
