- 二期优化：一期不再强调“内置窗口”口径，客户可见文案统一为“平台授权页面 / 平台采集页 / 当前可见评论”。
- 二期优化：本地网页版调试优先，不需要打包；作品评论采集主流程确定为“边采集边回复”：打开作品后按关键词采集当前评论，命中后立即进入精准评论回复队列；已回复成功和未回复成功都保留到执行/完成卡片，用于复盘，不再内置关注、私信或二次跟进链路。表单默认边采集边回复、全量采集上限 500，并支持“采集并精准回复”。
- 二期界面收敛：不再把所有一期隐藏能力全部放开；当前主导航回到简洁主流程，只保留今日工作台、账号池、Agent 执行和 Agent 执行下面的结果复盘。评论池、采集任务、后台日志、诊断、模型设置、导入规则和审计等作为内部能力隐藏。
- 抖音精准评论回复路径：作品链接可使用 `modal_id` 链接；进入作品后先按 `X` 打开评论区，再按关键词匹配评论内容，点击该评论下方“回复”，停留等待回复框切换完成，必须确认输入框上下文出现 `回复@目标用户: 目标评论内容` 后，才在底部“留下你的精彩评论吧”输入回复，并点击红色发送箭头或回车发送。
- 抖音关键词回复实测规则：不能只按第几个“回复”按钮或视觉位置发送；每次点击回复后都要二次读取上下文，若 `回复@` 用户或评论片段不一致，立即停止本条发送并重新定位。已在 `server.js` 的精准评论回复链路增加抖音回复框目标校验，防止错挂到相邻评论。
- 抖音关键词回复实测结果：在当前作品 `7614022015325949230` 中，以关键词“抖音精选”命中并成功回复 3 条，回复内容为“我也是”；目标分别为 `船长MC`、`七匹狼`、`小鱼肉`，每条发送前均确认 `回复@目标用户` 和评论内容，发送后均出现“已发布”。
- 抖音精准回复复测结果：在作品 `7645241069826772224` 中，以关键词“中国”命中 `幸福常在dl25` 的评论“这小伙子地道的中国通呀。”，系统成功定位原评论下方回复入口，写入 `回复@` 编辑器，并通过回车发送 `我也是`；接口记录 `sent=1 / failed=0`，发送策略为 `enter-inline-precise-reply`。
- 抖音精准回复稳定性修正：真实发送路径已固定为“原生回复元素点击 -> 当前活动 `回复@` 编辑器验证 -> 清空旧草稿 -> 写入话术 -> 活动编辑器复查 -> 回车发送”。禁止回退到作品主评论框强写，未确认 `回复@` 状态时不发送。
- 抖音作品扫描切换修正：关键字精准回复成功后，不再因为已达到本批回复上限就立刻切换作品；系统会继续滚动扫描当前作品评论到底，命中评论继续去重入池，额度用完后不再超发，待当前作品扫描完成后再返回外层切换下一条作品。
- 抖音/小红书实测确认口径：切号等待默认 5 秒，避免等待过长；进入作品后从第一条评论开始采集，精准评论命中即回复；回复成功后继续扫描当前作品全量评论到底；没有精准评论命中或需要首评兜底时，才对作品主评论/首评执行回复。该流程在抖音和小红书两端保持一致。
- 抖音关键词回复待优化/已修正方向：实测确认切换账号回复可成功，但需要提高全量覆盖和连续命中稳定性。作品直达模式应先深扫当前作品最多 500 条可见评论，并按“平台用户 + 评论内容”去重后再生成回复队列，避免重复 DOM 行占用回复名额；若相邻两条评论都命中关键词，必须分别入队、分别确认 `回复@目标用户`，不能因为第一条成功或失败就漏掉第二条。执行速度可适度降低默认间隔，但仍要保留回复框上下文停留确认。
- 抖音作品直达回复修正：全量爬取后逐条回复时，不应每回复一条就重新刷新作品页。当前已改为同一作品页内连续定位和回复：首次打开作品并采集评论，后续命中评论复用当前真实浏览器页面，只重新确认评论区和回复框上下文，减少平台风控触发概率。
- 全量爬取沉淀修正：作品直达关键字精准回复时，全量扫描只用于找全命中评论；只有命中关键词的去重评论会写入评论池并进入精准回复队列，未命中的评论不保留。任务复盘文案会显示命中评论入池数量、重复数量，并在结果复盘里区分已成功和未成功。
- 未成功二次操作：结果复盘已新增“批量重试未成功”，并在单条操作列提供“重新自动回复 / 打开来源人工处理 / 标记已处理 / 忽略”。批量重试默认取当前筛选范围内最多 20 条未成功线索，按平台分组，抖音和小红书可并发，各平台内部串行；自动回复复用线索保存的来源作品、评论作者、评论内容和原回复话术，并调用精准评论回复链路。
- 小红书精准评论回复路径：复制链接常见格式为 `https://www.xiaohongshu.com/explore/笔记ID?xsec_token=...&source=web_explore_feed`，系统应保留 `xsec_token`、`xsec_source`、`source`、`channel` 等访问参数并识别为具体笔记链接。进入笔记页后，按关键词匹配评论内容，点击目标评论下方的消息气泡或“回复”，停留等待底部输入框切换；优先用“刚点击的回复按钮附近上下文”确认作者和评论片段，底部输入框只确认已进入回复/发送态，因为小红书底部 `contenteditable` 的父级 DOM 会混入整页评论文本。确认目标一致后输入话术，最后回车或点击“发送”。若回复框上下文与命中评论不一致，停止本条并重新定位。
- 小红书页面实测结果：在用户打开的笔记 `69f455fc000000003601b011` 中，点击可见评论下方“回复”后会展开该评论回复区，并在底部出现 `contenteditable` 输入框、`发送` 和 `取消`；输入草稿可成功写入，点击取消后草稿消失。注意取消后页面仍可能保留“发送/取消”回复面板状态，后续判断成功闭环不能只看按钮存在，必须结合输入框内容清空或平台发送确认。

## DeepSeek 接入说明

- 页面入口：「发现」页 ->「AI 意图识别」
- 默认接口：`https://api.deepseek.com`
- 默认模型：`deepseek-chat`
- 也可通过环境变量配置：`DEEPSEEK_API_KEY`、`DEEPSEEK_MODEL`、`DEEPSEEK_BASE_URL`
- 当前实现：本地规则先筛，未命中的评论再交给 DeepSeek 复核
- 注意：不要把真实 API Key 写进源码、README 或安装包

## 重要文件

- `server.js`：后端接口、授权、规则发现、DeepSeek 调用、线索生成
- `index.html`：前端单页 UI、矩阵入口、采集方式、AI 配置、线索跟进
- `electron-main.js`：桌面壳启动本地服务
- `data/seed.json`：初始化数据
- `data/mvp-db.json`：当前运行数据
- `scripts/generate-license.js`：授权码生成
- `scripts/make-icon.js`：图标生成
- `docs/licensing-and-signing-plan.md`：授权、签名和长期规划

## 下一轮建议优先级

当前进入二期优化节奏，优先保证安装包真实可用、核心采集稳定、线索跟进闭环清晰，再逐步上云端、多账号和企业通知能力。

1. 先把本地网页版固定端口调试走顺：围绕“边采集边回复 + 精准评论回复 + 执行/完成卡片”做 Agent 执行闭环，不再扩散页面入口。
2. 验证精准评论回复链路可用性：边采集边回复、关键词筛选命中评论、打开来源作品、定位评论人/评论内容、在该评论下方挂载回复、发送前人工确认、成功/失败完成卡片和账号额度消耗。
3. 做作品评论采集稳定性：评论弹层、滚动加载、重复评论过滤、关键词命中、失败提示和采集数量达标。
4. 做真实平台授权或官方接口评估，把当前平台页面可见采集升级为更稳定的生产采集接入，并优先确认平台允许范围内的评论回复能力。
5. 做线索合并二期：同一用户跨作品、跨平台的人工合并和拆分。
6. 做企业微信或内部通知：候选线索命中和回复失败时推送运营人员复核。
7. 做账号绑定二期：把可视化授权检测扩展到账号级采集/回复排队、授权过期提醒、IP 配置、账号健康百分比和平台差异化检测。
8. 继续做云端多租户迁移：抽象数据访问层、MySQL Schema、租户授权后台、套餐额度和平台管理员后台。
9. 做 Windows 签名落地：购买代码签名证书后开启 electron-builder 签名流程。
10. 手机采集另开分支承接：再验证安卓测试机、手机助手、可见评论整理和导入评论池流程。

## 风险边界

产品应继续保持：只处理公开可见、用户授权、官方留资、自有粉丝或人工导入数据；不做验证码绕过、风控对抗、逆向私有接口、无上限批量私信或非必要个人信息采集。官方号/企业号触达可以做，但必须是平台允许范围内的小范围、限频、可暂停、可审计触达。
